# README 维护

## 概述

README.md 是项目的首页，是绝大多数人接触 Xyncra 的第一份文档。README 的质量直接影响项目给人的第一印象、采用率和贡献者参与度。

本文档定义 README.md 的定位、内容和维护规范。注意：README 不等于 Wiki，README 是入口，Wiki 是深度知识库。

---

## 目标受众

### 主要读者

README 需要同时服务以下四类读者，每一类都在 README 中寻找不同的信息：

| 读者类型 | 目标 | 关注内容 |
|----------|------|----------|
| **潜在用户** | 评估是否使用 Xyncra | 功能列表、相比其他方案的差异、快速上手成本 |
| **新用户** | 快速部署和运行 | 快速开始、配置参数、客户端使用 |
| **贡献者** | 决定是否参与开发 | 项目结构、开发流程、贡献指南 |
| **技术评估者** | 评估技术方案 | 架构概览、协议设计、技术栈 |

### 阅读路径

```
第一次访问 README 的典型阅读路径：

潜在用户 → 项目简介 → 功能亮点 → 架构概览 → 快速开始
  (30秒)     (10秒)     (20秒)     (30秒)     (决定是否继续)

贡献者   → 项目简介 → 项目结构 → 开发命令 → 贡献指南
  (30秒)     (10秒)     (15秒)     (10秒)     (决定是否开始)
```

README 要在 30 秒内让读者理解"这是什么项目、它解决什么问题、我是否应该继续了解"。

### 非目标读者

以下人群不应被 README 服务。他们有更专门的入口：

- **需要深度理解系统的开发者** → 架构视角 Wiki
- **遇到问题的运维人员** → 运维视角 Wiki 和具体文档
- **编写测试的 QA** → 测试视角 Wiki
- **配置 Agent 的 AI 开发者** → Skill 视角 Wiki

README 不试图满足所有人的所有需求。它的职责是**引导到正确的入口**。

## README 内容结构

### 必须包含的内容

以下是 README 必须包含的章节，按推荐顺序排列：

#### 1. 项目标识

- 项目名称和一行标语（Tagline）
- 3-5 个关键特性的标签式摘要

```
# Xyncra Server

**A distributed messaging backend with built-in AI agent support.**

> Bidirectional RPC over WebSocket · Multi-device sync · Offline resilience · Streaming · Human-in-the-Loop
```

#### 2. 项目简介（Why）

回答 "为什么要有这个项目"——它解决什么问题，与竞品有何不同。

形式：对比表格或问题-方案对照。这部分不展示细节，只给出概览和差异点。

#### 3. 架构概览（How）

使用 ASCII art 展示系统的高层架构图。图中只出现核心组件和连接方式。不加细节说明——细节留给 Wiki。

配合 3-5 句话说明各层职责。每层一句话，不超过 15 个字。

#### 4. 功能列表（What）

分类列出核心功能，每个功能带一句话说明。

```
### 💬 Messaging Core

- **Bidirectional RPC** — Both client and server can initiate calls...
- **Persistent updates** — Sequence-numbered update log...
```

分类标准：

| 类别 | 说明 |
|------|------|
| Messaging Core | 消息收发、同步、多设备 |
| AI Agent Runtime | Agent 定义、流式输出、HITL、工具调用 |
| Infrastructure | 部署、存储、可观测性 |

#### 5. 快速开始（How to start）

目标：让用户从零到运行不超过 5 分钟。

内容：
- 前置依赖（Go 版本、Redis）
- 构建命令
- 运行命令
- 可选：Docker 部署

只包含最简路径。更多配置项和部署选项见 Wiki。

#### 6. CLI 客户端（可选但推荐）

如果项目有 CLI 客户端，展示最常用的命令。每个命令附带一句话说明和输出示例。

#### 7. 协议概览（可选）

如果 README 面向协议开发者，可以包含协议的简要说明。协议细节链接到 API 参考文档。

#### 8. 项目结构（Project Structure）

展示顶层目录结构。解释每个目录的职责。不需要深入到子目录层级，除非特别重要。

#### 9. 开发命令

开发相关命令（测试、格式化、lint）。用 Makefile 封装，README 只列出顶层命令。

#### 10. 文档索引

列出关键的文档入口。包括 Wiki 的简要说明和链接，以及 docs/ 目录中的重要文档。

#### 11. 贡献指南

简要说明如何贡献。包括：
- 报告 Bug 的方式
- 提交 PR 的流程
- 需要遵循的规范

#### 12. License

项目的开源许可证信息。

### 不应包含的内容

README 不应该包含以下内容：

| 内容 | 不应包含的原因 | 应该放在哪里 |
|------|---------------|-------------|
| 完整的 API 参数表 | 篇幅太长，且变更是常态 | `docs/API.md` 或 Wiki Onboarding |
| 所有配置项和默认值 | 同样太长，变化频繁 | Wiki Onboarding - 配置说明 |
| 详细的架构设计理由 | 目标读者不需要深入了解 | Wiki Architecture - 设计决策 |
| 测试策略和测试用例 | 只有测试人员关心 | Wiki Testing |
| 部署拓扑和运维手册 | 只有运维人员关心 | Wiki DevOps |
| 完整的产品决策清单 | 篇幅太长 | `docs/decisions/` |
| 所有的 Agent 配置示例 | 空间有限 | Wiki Skills - Agent 开发 |
| 常见问题排查 (FAQ) | 会随版本增长，不易维护 | Wiki 对应视角或独立 FAQ 文档 |
| 团队信息和联系方式 | 人员变动频繁 | GitHub 组织页面或 CONTRIBUTORS |
| 贡献者列表 | 自动更新更好 | GitHub 的 Contributors 功能 |
| Roadmap 和计划 | 计划会变，产生误导 | GitHub Projects 或 Issues |

### 核心原则

```
README 是入口，不是百科全书。
每个章节应当能在 30 秒内阅读完毕。
链接到细节，而非堆砌细节。
```

更具体的指导：

**长度为王**：README 保持在 400-600 行是一个健康的范围。超过 600 行应该考虑将内容移到 Wiki。

**渐进披露**（Progressive Disclosure）：读者看到的第一个代码块应该是 `make build` 或 `docker compose up` 级别的简单操作。深入到第二层知识时才需要链接。

**链接策略**：README 中的链接指向 Wiki 和 docs/ 中的具体文档。每条链接应说明读者会在目标文档中找到什么。

```
📖 Full protocol specification: [docs/API.md](docs/API.md)
📖 Configuration details: [Configuration Guide](wiki/onboarding/configuration.md)
```

## README 与 Wiki 的关系

### 职责分工

```
                  README                        Wiki
───────────────┬────────────────────    ─────────────────────
读者           │ 所有人（第一接触点）    定向读者（深入需求）
信息密度       │ 低（概要）             高（深度）
更新频率       │ 低（大版本变更）        持续（随代码变更）
篇幅           │ 400-600 行             不限
目的           │ 让读者决定"要不要用"    让读者知道"怎么用"
内容形式       │ 概览 + 示例 + 链接      深度说明 + 细节
```

### 链接规则

README 到 Wiki 的链接规则：

1. **从概览到细节**：README 中的每条说明在 Wiki 中都有对应的详细文档
2. **双向指向**：Wiki 中的某些文档应该反向指向 README（特别是快速入门部分）
3. **使用简短说明**：链接文本包含简短的说明，告诉读者链接通向哪里

**正确示例**：
```
详见 [系统架构](wiki/architecture/system-architecture.md) 了解组件关系和分层设计
详见 [快速开始](wiki/onboarding/quick-start.md) 了解更详细的配置选项
```

**错误示例**：
```
详见 [Wiki](wiki/)
详见 [这里](wiki/architecture/system-architecture.md)
```

### 内容重叠管理

README 中的架构概览是 Wiki 中系统架构文档的**一句话版本**。如果 Wiki 中的架构图更新了，README 中的架构图也必须同步更新。具体同步规则见下文"更新触发器"。

## 多语言 README

### 策略选择

Xyncra 选择**单语言 README** 策略：默认使用英文书写 README。

原因：

1. **开源项目的通用语言**：英文是开源社区的事实标准
2. **维护成本**：多语言 README 需要同时维护多份文档，容易不同步
3. **目标受众**：项目期望吸引全球贡献者

### 如果未来需要多语言

如果未来决定支持多语言 README，遵循以下规范：

1. **主 README 为英文**：`README.md` 始终使用英文
2. **语言标识**：其他语言使用后缀：`README.zh-CN.md`、`README.ja-JP.md`
3. **语言选择器**：主 README 顶部使用语言切换列表：

```markdown
# Xyncra Server

[中文](README.zh-CN.md) | [日本語](README.ja-JP.md)
```

4. **同步策略**：

| 级别 | 要求 | 说明 |
|------|------|------|
| L1 关键信息 | 所有语言必须同步 | 项目简介、架构概览、快速开始 |
| L2 次要信息 | 主语言优先，其他语言延迟同步 | 功能列表、CLI 使用 |
| L3 详细信息 | 仅主语言维护 | 协议细节、项目结构 |

5. **自动化检查**：CI 中检查翻译版本的完成率，低于 80% 发出警告

## 更新触发器

### 必须更新 README 的事件

以下事件发生时，必须检查和更新 README：

| 事件 | 需要更新的章节 | 时间要求 |
|------|---------------|----------|
| 新增核心功能 | 功能列表、架构图（如适用） | 随功能 PR 一起提交 |
| 修改快速开始流程 | 快速开始、前置依赖 | 随流程变更 PR 提交 |
| 修改架构设计 | 架构概览、架构图 | 随架构变更 PR 提交 |
| 修改协议 | 协议概览（如适用） | 随协议变更 PR 提交 |
| 修改项目目录结构 | 项目结构 | 随结构变更 PR 提交 |
| 修改构建/开发命令 | 开发命令 | 随命令变更 PR 提交 |
| 增加或修改 CLI 功能 | CLI 客户端 | 随 CLI 变更 PR 提交 |
| 新增或删除 Wiki 视角 | 文档索引 | 随视角变更 PR 提交 |

### 不需要更新 README 的事件

| 事件 | 不需要的原因 |
|------|-------------|
| 内部重构（不影响外部接口） | README 面向的是外部读者 |
| 修复 Bug（不影响功能清单） | README 不记录 Bug 信息 |
| 新增依赖库 | README 不列举所有依赖 |
| 优化性能 | README 不记录性能指标 |
| 增加新的单元测试 | README 不记录测试详情 |
| 修改内部注释 | 注释是代码的一部分 |

### 维护检查流程

每次修改 README 时，使用以下检查清单：

- [ ] 所有内部链接是否有效？
- [ ] 代码示例是否可运行？
- [ ] 架构图是否与当前代码一致？
- [ ] 功能列表是否是最新的？
- [ ] 快速开始是否覆盖了最简路径？
- [ ] 配置参数是否有变动？
- [ ] 是否保持了 400-600 行的篇幅范围？
- [ ] 是否有新的 Wiki 视角需要添加到文档索引？

### 同步规则

当 Wiki 内容变更影响 README 时，使用以下同步规则：

```
Wiki 变更 → README 是否需要同步？
   ├─ Wiki 新增主题文档 → 通常不需要（README 不列举所有主题）
   ├─ Wiki 主题文档重写 → 检查 README 中是否有对应概述
   │   ├─ 有 → 同步更新 README 中的概述
   │   └─ 无 → 不需要操作
   ├─ Wiki 删除主题文档 → 检查 README 中是否有指向该文档的链接
   │   ├─ 有 → 更新或移除链接
   │   └─ 无 → 不需要操作
   └─ Wiki 变更架构设计 → 同步更新 README 中的架构概览和架构图
```

### PR 中 README 的审查

每个 PR 在 Review 阶段，Reviewer 应当评价：

1. 代码变更是否影响到 README 中的任何内容
2. 如果影响，README 是否已同步更新
3. 如果 PR 包含了 README 更新，更新是否准确

在 CI 中可以添加检查：当特定文件发生变更时，自动标记 README 需要审查。

## README 的 Markdown 风格

### 风格约定

与 Wiki 文档一致的 Markdown 风格：

- 标题使用 ATX 风格（`# `）
- 章节之间使用 `---` 分隔线
- 表格使用 GitHub Flavored Markdown 格式
- 代码块标注语言类型
- 内联代码和路径使用反引号

### README 特有的风格

- 使用 emoji 作为功能分类的视觉标记（`### 💬 Messaging Core`）
- 使用引用块 `>` 突出关键特性
- 使用对比表格展示差异化优势
- 使用连续短横线 `---` 分隔大章节

### 链接检查

定期（每月或每个发布周期）检查 README 中的所有链接是否有效。包括：

- 内部文件链接（文档、代码）
- 外部链接（GitHub、第三方资源）
- Wiki 链接是否仍然指向存在的文档

## 备选方案：CONTRIBUTING.md

当 README 的"贡献指南"章节增长到超过 50 行时，应拆分为独立的 `CONTRIBUTING.md` 文件。README 中的贡献指南保持简短，然后通过链接指向 `CONTRIBUTING.md`。

`CONTRIBUTING.md` 的内容：

- 详细的 PR 提交流程
- 代码规范要求
- 测试要求
- Review 流程说明
- Code of Conduct

---

## 备选方案：CHANGELOG.md

README 不包含版本变更历史。项目应该有一个独立的 `CHANGELOG.md`（或使用 GitHub Releases）记录版本变更。

`CHANGELOG.md` 的内容规范（遵循 [Keep a Changelog](https://keepachangelog.com/)）：

```
# Changelog

## [1.2.0] - 2026-07-15

### Added
- 新功能描述

### Changed
- 变更描述

### Fixed
- Bug 修复描述
```

README 中应该在文档索引或快速开始附近包含 Changelog 的链接。
