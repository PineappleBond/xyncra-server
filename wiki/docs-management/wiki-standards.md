# Wiki 规范

## 概述

Xyncra 采用**多视角知识库**（Multi-Perspective Knowledge Base）架构组织项目文档。与传统的单一大文档或按模块划分的文档体系不同，多视角知识库围绕**读者角色**和**关注维度**组织内容，每个视角独立成目录，包含索引和若干主题文档。

本文档定义 Wiki 的目录结构、文件命名、索引要求、交叉引用规则和维护工作流。

---

## 多视角架构

### 视角定义

视角（Perspective）是 Wiki 的一级组织单位，每个视角对应一个特定的读者角色或关注维度。当前定义 9 个视角：

| 视角目录 | 目标读者 | 关注的问题 |
|----------|----------|-----------|
| `architecture/` | 架构师、全体开发者 | 系统怎么设计的？为什么这么设计？ |
| `development/` | 开发者 | 怎么写代码？规范是什么？ |
| `testing/` | 测试人员、开发者 | 怎么测试？测试策略是什么？ |
| `onboarding/` | 外部接入者、新用户 | 怎么接入？配置是什么？ |
| `docs-management/` | 文档维护者 | 文档怎么写？规范是什么？ |
| `skills/` | AI 辅助开发者 | 怎么用 Claude Code？Agent 怎么开发？ |
| `devops/` | 运维人员、开发者 | 怎么部署？怎么运维？ |
| `observability/` | 开发者、SRE | 怎么监控？怎么排查问题？ |
| `dependencies/` | 架构师、开发者 | 依赖哪些库？为什么选择它们？ |

### 视角创建原则

新增视角必须满足以下条件：

1. **有明确的读者群体**：该视角的存在是因为有一类读者需要从特定角度理解项目
2. **有 3 个以上的候选主题**：新视角至少要有 3 个以上可预见的话题文档
3. **不与其他视角重叠**：如果话题可以归入已有视角，优先扩展已有视角而非新建
4. **有维护者**：每个视角必须有明确的维护者（可以是多人）

不满足上述条件时，优先将内容放入已有视角的合适主题文档中。

### 目录结构

每个视角目录的结构固定为：

```
wiki/<perspective>/
├── index.md          # 索引文件（必需）
├── topic-1.md        # 主题文档
├── topic-2.md
└── ...
```

根目录 `wiki/index.md` 是所有视角的总索引。

## 目录命名规范

### 视角目录命名

- 使用**英文小写**，多个单词以连字符 `-` 分隔
- 名称应准确反映视角内容，优先使用单数名词
- 不允许使用缩写（除非是行业通用缩写，如 `devops`）

**正确示例**：
- `architecture/`
- `docs-management/`
- `observability/`

**错误示例**：
- `Architecture/`（大写）
- `doc_mgmt/`（下划线 + 缩写）
- `docs/`（过于通用，与已有 `docs/` 目录混淆）

### 主题文档命名

- 使用**英文小写**，多个单词以连字符 `-` 分隔
- 文件名应准确概括文档主题
- 避免编号前缀（除非文档之间有严格的顺序依赖）
- 避免使用 `README.md`、`index.md` 之外的特殊文件名
- 避免过长的文件名（建议不超过 40 个字符）

**正确示例**：
- `system-architecture.md`
- `coding-standards.md`
- `client-usage.md`

**错误示例**：
- `01-system-architecture.md`（编号前缀，除非有顺序依赖）
- `README.md`（与视角目录内文件语义冲突）
- `关于消息发送流程的详细说明.md`（中文文件名）
- `how-to-set-up-the-development-environment-and-run-tests.md`（过长）

### 索引文件命名

每个视角目录下的索引文件必须命名为 `index.md`。根目录索引文件同样命名为 `wiki/index.md`。

## 索引文件要求

### 结构模板

每个视角的 `index.md` 必须包含以下部分：

```markdown
# <视角中文名称>

<一句话说明视角关注的内容和范围>。

## 主题文档

| 文档 | 说明 |
|------|------|
| [文档标题](topic-file.md) | 一句话说明文档内容 |
| ... |

## 核心原则

<2-5 条该视角的核心原则，每条一行>

1. <原则 1>
2. <原则 2>
```

### 内容要求

1. **标题**：使用一级标题 `#`，中文名称
2. **视角说明**：1-2 句话，说明该视角关注什么、为谁服务
3. **主题文档表**：使用 Markdown 表格，列出所有主题文档及其一句话说明
   - 第一列是文档链接（`[标题](路径)`）
   - 第二列是说明（一句话）
4. **核心原则**：3-5 条该视角的核心原则，体现该视角的价值观

### 根索引要求

`wiki/index.md` 是所有视角的总索引，必须包含：

1. 项目 Wiki 的整体说明
2. 完整的视角列表（包含说明和目标读者）
3. 通用的文档库规范

### 索引维护规则

- 新增主题文档时必须同步更新 `index.md`
- 重命名或删除主题文档时必须同步更新 `index.md`
- index.md 本身不包含详细的技术内容，只做导航
- 每个索引文件控制在 20-30 行，保持简洁

## 交叉引用规则

### 引用格式

跨视角引用使用相对路径 Markdown 链接：

```markdown
详见 [系统架构](../architecture/system-architecture.md)

参见 [编码规范 - 命名约定](../development/coding-standards.md#命名约定)
```

### 引用场景

以下场景必须使用交叉引用：

1. **依赖说明**：一个主题依赖另一个视角的知识
   - 例：测试文档引用架构文档中的协议说明
   - `详见 [协议设计](../architecture/protocol-design.md)`

2. **上下文补充**：文档需要外部背景知识才能理解
   - 例：开发指南引用部署文档
   - `部署环境配置见 [环境配置](../devops/environment-config.md)`

3. **决策溯源**：代码实现引用 ADR 决策
   - 例：Handler 实现引用架构决策
   - `该设计决策详见 [ADR-002](../architecture/design-decisions.md#adr-002为什么选择-websocket而非httpgrpc)`

4. **避免重复**：同一信息在多个视角中出现时，一个为主，其他引用
   - 例：API 参数说明在 `onboarding/` 中详细描述，测试文档中只引用不重复

### 引用规范

- 使用**描述性链接文本**，不要使用"点击这里"、"参考链接"等模糊表述
- 链接文本应描述被引用文档的内容
- 引用时附带简要说明，帮助读者判断是否需要跳转
- 同一页面的引用使用锚点定位到具体章节
- 跨文件引用使用相对路径，以 `wiki/` 为基准

**正确示例**：
```markdown
消息的序列号分配机制参见 [协议设计 - 序列号策略](../architecture/protocol-design.md#序列号策略)
序列号从 1 开始递增，每对 (user_id, conversation_id) 独立计数。
```

**错误示例**：
```markdown
详情见[这里](../architecture/protocol-design.md)
```

### 双向链接

当两个文档之间存在较紧密的关联时，应在两个文档中互相引用：

```markdown
文档 A：消息的持久化策略见 [数据流 - 先落库再处理](../architecture/data-flow.md)
文档 B：发送消息的整体流程见 [开发者指南 - 消息发送](../development/developer-guide.md#消息发送)
```

### 引用更新

当被引用文档发生以下变化时，必须检查并更新所有引用点：

1. 文档重命名或移动
2. 章节结构调整（锚点失效）
3. 关键信息变更（引用内容不再准确）

## 文件元信息

### YAML Front Matter（可选）

对于较长的主题文档，可以在文件顶部使用 YAML Front Matter 记录元信息：

```markdown
---
title: 系统架构概览
perspective: architecture
updated: 2026-07-15
reviewer: user
maintainer: user
status: stable # draft | stable | deprecated
---
```

### 字段说明

| 字段 | 必填 | 说明 |
|------|------|------|
| `title` | 是 | 文档标题，与 H1 保持一致 |
| `perspective` | 是 | 所属视角目录名 |
| `updated` | 是 | 最后更新日期（YYYY-MM-DD） |
| `reviewer` | 否 | 最近一次审阅者 |
| `maintainer` | 否 | 文档维护者 |
| `status` | 是 | 文档状态：`draft`（草稿）、`stable`（稳定）、`deprecated`（废弃） |

### 索引文件要求

索引文件的 Front Matter 可选，但 `wiki/index.md` 必须包含 `updated` 字段。

## 文档状态管理

### 状态定义

| 状态 | 含义 | 预期行为 |
|------|------|----------|
| `draft` | 初始草稿，内容可能不完整或未审阅 | 可被搜索到，但标记为"草稿" |
| `stable` | 内容经过审阅，与当前代码一致 | 正常展示，作为可信参考 |
| `deprecated` | 文档描述的功能已废弃或已被替代 | 保持可见但添加废弃警告，引导读者到替代文档 |

### 状态转换

```
draft → stable（审阅通过）
stable → deprecated（功能废弃或重构）
deprecated → （文档清理，删除或归档）
```

### 废弃文档处理

1. 在文档顶部添加警告框：
   ```markdown
   > ⚠️ **废弃**：本文档描述的功能已不再维护。请参考 [替代文档](../path/to/replacement.md)。
   ```
2. 在对应的 `index.md` 中保留条目但添加 `（已废弃）` 标记
3. 废弃文档保留至少一个发布周期后再删除
4. 删除前确认无任何交叉引用指向该文档

## 维护工作流

### 日常维护

| 触发条件 | 操作 | 负责人 |
|----------|------|--------|
| 新增功能 | 添加或更新相关主题文档 | 功能开发者 |
| 重构代码 | 同步更新相关文档 | 重构者 |
| 修复 Bug | 视情况更新相关文档 | Bug 修复者 |
| 代码 Review | Reviewer 检查文档是否需要更新 | Reviewer |

### 定期维护

| 频率 | 任务 | 说明 |
|------|------|------|
| 每周 | 检查文档状态 | 确认所有文档状态与代码一致 |
| 每月 | 清理废弃文档 | 删除或更新已废弃的文档 |
| 每个发布周期 | 全面审阅 | 审阅所有视角的文档完整性和准确性 |

### 维护责任矩阵

| 视角 | 主要维护者 | 审阅者 |
|------|-----------|--------|
| `architecture/` | 架构师 | 全体开发者 |
| `development/` | 所有开发者 | 团队 Lead |
| `testing/` | 测试负责人 | 开发者 |
| `onboarding/` | 项目维护者 | 新用户 |
| `docs-management/` | 文档维护者 | 团队 Lead |
| `skills/` | AI 工具负责人 | 开发者 |
| `devops/` | DevOps 负责人 | 运维 |
| `observability/` | SRE 或架构师 | 开发者 |
| `dependencies/` | 架构师 | 开发者 |

### 文档修改流程

1. **识别需求**：确定需要修改的文档和视角
2. **修改内容**：按本文档规范修改
3. **本地预览**：确保 Markdown 渲染正确
4. **交叉引用检查**：更新所有相关引用
5. **索引同步**：确认 `index.md` 已更新
6. **提交审阅**：作为 PR 提交，附带文档变更说明
7. **合并发布**：审阅通过后合并

### 批量更新场景

当发生以下大规模变更时，需要对全 Wiki 进行批量检查和更新：

- 项目重命名或迁移仓库
- 核心协议或 API 变更
- 依赖库发生 breaking change
- 目录结构大规模调整

批量更新流程：

1. 使用 `grep` 或脚本搜索所有涉及变更加关键字
2. 逐文档检查并更新
3. 确认所有交叉引用链接有效
4. 更新所有受影响视角的 `index.md`

## 文档质量检查清单

提交文档变更前，使用以下清单自检：

### 内容完整性

- [ ] 文档有明确的标题和一句话说明
- [ ] 内容紧扣主题，不偏离文档标题
- [ ] 代码示例可运行且与实际代码一致
- [ ] 没有遗留的 TODO 或占位符

### 格式规范

- [ ] 使用正确的 Markdown 语法（标题层级、表格、代码块）
- [ ] 代码块标注了正确的语言（`go`、`bash`、`json`、`yaml` 等）
- [ ] 表格有对齐的头部分隔线
- [ ] 文件以换行符结尾

### 链接完整性

- [ ] 所有内部链接可通过相对路径访问
- [ ] 锚点指向正确的章节
- [ ] 外部链接可访问
- [ ] 没有死链或循环引用

### 视角一致性

- [ ] 内容属于当前视角的范畴
- [ ] 没有与其他视角主题文档的明显重复
- [ ] 如果引用了其他视角的内容，使用了交叉引用而非重复复制

## 与 `docs/` 目录的关系

项目历史原因存在 `docs/` 目录（包含 API 参考、产品决策、设计文档等）。`wiki/` 与 `docs/` 的关系：

| 维度 | `wiki/` | `docs/` |
|------|---------|---------|
| 组织方式 | 多视角知识库 | 按文件类型/功能分类 |
| 内容粒度 | 单主题单文件 | 大文档集中 |
| 维护方式 | 持续维护 | 部分文档已过时 |
| 目标读者 | 按视角区分 | 混合读者 |
| 状态 | 活跃维护 | 逐步迁移到 `wiki/` |

迁移策略：

1. 新写文档优先放入 `wiki/` 对应视角
2. `docs/` 中仍有价值的内容按计划迁移到 `wiki/`
3. 迁移完成后在 `docs/` 中添加 README 说明指向 `wiki/`
4. `docs/API.md` 作为协议参考保留（适合作为单一权威文档）
5. `docs/decisions/` 中的产品决策保留（与 `wiki/architecture/design-decisions.md` 互补）

## 文件编码和行尾

- 所有文档使用 **UTF-8** 编码
- 行尾使用 **LF**（Unix 风格）
- 每行不超过 **120 个字符**（表格和链接可例外）
- 文件末尾保留一个空行
