# Vibe Coding 指南

## 什么是 Vibe Coding

Vibe Coding 是 Xyncra 项目中采用的一种 AI 辅助开发实践——开发者用自然语言描述意图，AI 生成代码，人工审查并集成。这种方法不是完全替代开发者，而是将开发者的角色从"编写每一行代码"转变为"定义意图、审查结果、做出决策"。

在 Xyncra 中，Vibe Coding 使用 Claude Code（opencode）作为主要 AI 辅助工具，结合项目自定义的 Skill 和 Command 体系。

## 开发工作流

### 日常开发循环

```
意图描述 → AI 生成 → 人工审查 → 集成测试 → 提交
```

1. **意图描述**：用自然语言向 AI 描述你要实现的功能或修复的问题。尽量提供：
   - 背景上下文（涉及哪个模块、为什么需要这个改动）
   - 验收标准（什么算"完成"）
   - 约束条件（性能要求、兼容性等）

2. **AI 生成**：AI 分析代码库，生成实现方案。此时 AI 会：
   - 搜索相关代码理解现有模式
   - 遵循项目约定（参考已有实现风格）
   - 生成完整的代码改动

3. **人工审查**：这是最重要的环节。你需要检查：
   - 逻辑是否正确
   - 是否遵循了项目的代码风格和约定
   - 是否有安全隐患
   - 是否有不必要的复杂性

4. **集成测试**：运行相关测试确保改动不破坏现有功能：
   ```bash
   make test          # 单元测试
   make test-all      # 全部测试（需要 Docker 环境）
   ```

5. **提交**：使用项目约定格式提交代码。

### 复杂任务的拆分策略

对于大型功能，采用分层策略：

1. **规划阶段**：使用 `xyncra-task-planner` Skill 将粗略需求转化为详细执行计划
2. **实施阶段**：按计划分步执行，每步完成后运行测试
3. **审查阶段**：使用 `requesting-code-review` Skill 进行代码审查

### 何时使用 /plan 模式

- 功能涉及多个模块修改
- 需要数据库 schema 变更
- 需要设计新的接口或协议
- 不确定最佳实现路径时

## Skill 与一次性代码

### 何时创建 Skill

Skill 是对 AI 行为的一组指令封装，用于复用常见的开发模式。以下情况应考虑创建 Skill：

- **重复超过 3 次**的任务模式
- **涉及多步骤**的开发流程（如 E2E 测试编写）
- **需要特定领域知识**的操作（如 Eino Agent 配置）
- **团队共享**的开发规范或流程

#### 判断树

```
这个操作会在未来重复出现吗？
├── 是 → 这个操作需要多步骤或特定领域知识吗？
│   ├── 是 → 创建 Skill
│   └── 否 → 创建 Command（单步快捷操作）
└── 否 → 直接用自然语言描述（一次性代码）
```

### 何时写一次性代码

- 简单的 bug 修复
- 实验性功能探索
- 仅用一次的脚本或工具
- 临时调试代码

### 现有 Skill 使用

项目已有的 Skill 见 [Skill 清单](skill-list.md)。在对话中使用 `@skill` 引用对应的 Skill 文件以激活其指令。

## Prompting 最佳实践

### 项目级规则（AGENTS.md）

项目的 `AGENTS.md` 定义了所有 AI 辅助开发的基础规则，包括：
- 交互语言（中文）
- 模糊需求的处理方式

AI 会在每次对话开始时自动加载这些规则。

### 有效 Prompt 的结构

一个高质量的 Prompt 应包含：

```
## 背景
[相关模块、上下文链接]

## 需求
[清晰描述你要什么]

## 约束
[性能要求、兼容性、安全考虑]

## 验收标准
[如何判断完成]
```

### 具体示例

#### 好的 Prompt

```
在 internal/handler/ 中新增一个获取用户在线状态的 handler。
- 通过 ConnectionStore 查询用户是否在线
- 返回 200 和 JSON {"online": true/false}
- WebSocket 方法名 system.get_presence
- 参考现有的 heartbeat.go 的实现风格
```

#### 不好的 Prompt

```
帮我加个在线状态功能
```

缺乏上下文、约束和验收标准，AI 需要来回追问才能理解你的真实需求。

### 上下文管理

- **提供文件路径**：引用具体代码文件而非模糊描述
- **使用 Git 信息**：涉及分支变更时提供 `git diff` 或 PR 链接
- **参考已有实现**："参考 XXX 的实现方式" 比单独描述更高效
- **避免过度上下文**：只提供相关文件，而非整个项目

### Agent 开发中的 Prompting

在开发 AI Agent（`agents/` 目录下的 YAML 文件）时，Agent 的 system prompt 应：

1. **明确角色定位**：Agent 是做什么的
2. **限定工具使用**：明确什么场景用什么工具
3. **定义边界**：什么不该做
4. **提供示例**：工具调用格式示例

参见 [Agent 开发指南](agent-development.md) 获得更多细节。

## 质量守门

### 自动检查

AI 生成代码后，应当运行：
- `make fmt` — 代码格式
- `make vet` — 静态分析
- `make test` — 单元测试

### 审查清单

在合入 AI 生成的代码前，逐项检查：
- [ ] 是否有不必要的依赖引入
- [ ] 是否有安全漏洞（SQL 注入、XSS、密钥泄露）
- [ ] 错误处理是否完整
- [ ] 日志是否恰当
- [ ] 测试是否覆盖边界情况
- [ ] 性能是否可接受
- [ ] 文档是否需要更新

## 常见陷阱

### 过度生成

AI 倾向于生成比需求更多的代码。当你发现 AI 在实现一些你没有要求的功能时，及时打断并重新聚焦。

### 幻觉依赖

AI 可能引入不存在的库或 API。审查时注意 `go.mod` 的变化，确保引入的依赖是真实且必要的。

### 上下文遗忘

长对话中 AI 可能忘记前文的约定。适时提醒 AI 查阅项目已有实现或 `AGENTS.md` 中的规则。

### 循环修复

AI 修复一个问题的同时可能引入新的问题。如果超过 3 轮修复仍未解决，考虑人工介入或重新描述问题。

### 幻觉上下文

AI 可能基于错误的假设生成代码。当 AI 说"我假设 XXX"时，应主动提供正确的上下文信息，而非让 AI 继续基于假设工作。

### 过度工程化

AI 倾向于为简单问题设计过度复杂的解决方案。对于简单的 CRUD 操作或配置变更，应当要求 AI 使用最简单直接的方式，而非设计抽象层和设计模式。

## 团队协作中的 Vibe Coding

### 代码一致性

当多个开发者使用 AI 辅助开发时，需要确保代码风格的一致性。方法包括：
- 共享 AGENTS.md 中的规则
- 使用 `make fmt` 统一格式化
- Code Review 时检查 AI 风格偏差

### 知识传递

AI 辅助开发可能导致"知其然不知其所以然"。建议：
- 关键决策留下注释或决策记录
- 定期进行非 AI 辅助的代码审查
- 团队分享 AI 使用经验

## Vibe Coding 的度量

### 效率度量

| 指标 | 目标 | 说明 |
|------|------|------|
| 意图到代码的时间 | < 5 分钟 | 从描述到生成可审查代码 |
| 代码审查通过率 | > 80% | 一次生成后审查通过的比率 |
| 测试覆盖率保持 | ≥ 当前水平 | AI 不应降低测试覆盖 |
| 修复轮次 | ≤ 3 | 一个功能最多 3 轮生成-审查循环 |

### 质量度量

- 生成的代码应通过 `make vet` 和 `make test`
- 不应引入未使用的依赖
- 遵循项目已有的代码风格和模式
