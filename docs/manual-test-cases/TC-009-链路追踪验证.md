# TC-009: 链路追踪验证

> **测试编号**: TC-009
> **测试类型**: 端到端集成测试（链路追踪）
> **覆盖范围**: OpenTelemetry 分布式链路追踪全链路验证
> **环境**: Docker Compose（Jaeger + xyncra-server）
> **最后更新**: 2026-07-17

---

## 1. 概述

本测试用例验证 Xyncra Server 的分布式链路追踪功能：从 WebSocket 接收到 Agent LLM 调用的完整链路，确保 Span 层级正确、属性完整、Token 消耗记录准确。

**测试目标**：
1. 验证 Jaeger 能够接收并展示追踪数据
2. 验证 Span 层级结构符合设计（ws.connection → ws.message.receive → handler.invoke → handler.broker.enqueue → mq.process → agent.execute → agent.llm.call）
3. 验证 LLM 调用的 Token 消耗记录在 `agent.llm.call` span 上
4. 验证跨 MQ 边界的 Trace Context 传播正确（同一个 trace 内）
5. 验证 Debug 采样策略对指定用户/设备强制采样
6. 验证 Tracing 关闭时无开销（no-op 保证）

**覆盖的关键决策**：
- 追踪 SDK: OpenTelemetry
- Exporter: OTLP/gRPC
- 默认后端: Jaeger All-in-One（Badger 嵌入式存储）
- 采样策略: 可配置比例 + DebugSampler（debug user/device 强制采样）
- MQ 传播: W3C Trace Context（同 trace，子 span）

## 4. 测试步骤

### 4.1 TC-009-1: 基础链路追踪验证

**目标**：验证 Jaeger 能接收追踪数据，Span 层级正确。

**步骤**：

1. 启动 Jaeger：
   ```bash
   docker compose -f deploy/docker-compose.yml up -d jaeger
   ```

2. 等待 Jaeger 健康检查通过：
   ```bash
   docker compose -f deploy/docker-compose.yml ps jaeger
   # 确认状态为 healthy
   ```

3. 启动 server（启用 tracing）：
   ```bash
   XYNCRA_TRACING_ENABLED=true \
   XYNCRA_TRACING_OTLP_ENDPOINT=localhost:4317 \
   XYNCRA_TRACING_OTLP_INSECURE=true \
   XYNCRA_TRACING_SAMPLING_RATE=1.0 \
   ./bin/xyncra-server
   ```
   或者使用 docker compose 一体启动：
   ```bash
   docker compose -f deploy/docker-compose.yml up -d
   ```

4. 启动 CLI 客户端并发送消息触发 Agent：
   ```bash
   ./bin/xyncra-client send \
     --user-id alice --device-id laptop \
     --server ws://localhost:8080/ws \
     --agent weather-bot \
     --message "What is the weather in Beijing?"
   ```

5. 打开 Jaeger UI：
   ```
   http://localhost:16686
   ```

6. 在 Jaeger UI 中：
   - Service 选择 `xyncra-server`
   - 点击 "Find Traces"
   - 找到刚才的 trace

**预期结果**：
- [ ] Jaeger UI 能搜索到 trace
- [ ] trace 中包含以下 Span 层级：
  ```
  ws.connection
  ├── ws.message.receive
  │   └── handler.invoke
  │       └── handler.broker.enqueue
  │           └── mq.process
  │               └── agent.execute
  │                   ├── agent.build
  │                   └── agent.run
  │                       ├── agent.llm.call
  │                       ├── agent.tool.call (如果 Agent 调用了工具)
  │                       └── agent.llm.call (第二次迭代，如果有)
  └── ws.message.send
  ```
- [ ] 每个 Span 都有正确的 `xyncra.*` 属性前缀

---

### 4.2 TC-009-2: Token 消耗记录验证

**目标**：验证 `agent.llm.call` span 上记录了完整的 Token 消耗信息。

**步骤**：

1. 在 TC-009-1 的基础上，找到包含 LLM 调用的 trace
2. 展开 `agent.run` → `agent.llm.call` span
3. 查看 Span 属性

**预期结果**：
- [ ] `agent.llm.call` span 包含以下属性：
  - `xyncra.llm.model`：LLM 模型名（如 `qwen3.7-plus`）
  - `xyncra.iteration`：迭代次数（`1`, `2`, ...）
  - `xyncra.llm.input_tokens`：输入 Token 数（如 `500`）
  - `xyncra.llm.output_tokens`：输出 Token 数（如 `200`）
  - `xyncra.llm.total_tokens`：总 Token 数（如 `700`）
  - `xyncra.duration_ms`：调用耗时（毫秒）
- [ ] Token 数均为正整数
- [ ] `total_tokens = input_tokens + output_tokens`

---

### 4.3 TC-009-3: 跨 MQ Trace Context 传播验证

**目标**：验证消息从 handler 入队到 MQ worker 处理，处于同一个 trace 中。

**步骤**：

1. 在 Jaeger UI 中找到触发 Agent 的 trace
2. 查看 `handler.broker.enqueue` span 和 `mq.process` span 的关系

**预期结果**：
- [ ] `mq.process` 是 `handler.broker.enqueue` 的子 span（或同一个 trace 中的后代 span）
- [ ] 两者共享相同的 Trace ID
- [ ] `mq.process` span 包含属性：
  - `xyncra.task.type`：任务类型（如 `agent_process`）
  - `xyncra.task.id`：任务 ID

---

### 4.4 TC-009-4: WebSocket 连接属性验证

**目标**：验证 `ws.connection` span 携带正确的用户/设备信息。

**步骤**：

1. 在 Jaeger UI 中找到 trace 的根 span `ws.connection`
2. 查看其属性

**预期结果**：
- [ ] `ws.connection` span 包含属性：
  - `xyncra.user.id`：用户 ID（如 `alice`）
  - `xyncra.device.id`：设备 ID（如 `laptop`）
  - `xyncra.connection.id`：连接 ID
- [ ] `ws.message.receive` span 包含属性：
  - `xyncra.method`：方法名（如 `send_message`）
  - `xyncra.size_bytes`：消息大小

---

### 4.5 TC-009-5: Debug 采样验证

**目标**：验证 Debug 用户对指定用户强制全量采样。

**步骤**：

1. 启动 server，配置低采样率但指定 debug 用户：
   ```bash
   XYNCRA_TRACING_ENABLED=true \
   XYNCRA_TRACING_OTLP_ENDPOINT=localhost:4317 \
   XYNCRA_TRACING_OTLP_INSECURE=true \
   XYNCRA_TRACING_SAMPLING_RATE=0.0 \
   XYNCRA_TRACING_DEBUG_USERS=alice \
   ./bin/xyncra-server
   ```

2. 用 `alice` 用户发送消息

3. 在 Jaeger UI 搜索 trace

**预期结果**：
- [ ] 虽然采样率为 0.0，但 `alice` 用户的 trace 仍然出现在 Jaeger 中
- [ ] `ws.connection` span 上可能有 `xyncra.debug` 标记

4. 用另一个用户（如 `bob`）发送消息

**预期结果**：
- [ ] `bob` 用户的 trace 不会出现在 Jaeger 中（因为采样率为 0 且不在 debug 列表中）

---

### 4.6 TC-009-6: Tracing 关闭时 No-op 验证

**目标**：验证 tracing 关闭时无开销、无数据发送。

**步骤**：

1. 启动 server，不启用 tracing：
   ```bash
   ./bin/xyncra-server
   # XYNCRA_TRACING_ENABLED 默认为 false
   ```

2. 发送消息触发 Agent

3. 打开 Jaeger UI 搜索 trace

**预期结果**：
- [ ] Jaeger UI 中搜不到任何 trace
- [ ] Server 启动日志中无 OTLP exporter 连接相关日志
- [ ] 业务功能正常（消息收发、Agent 执行不受影响）

---

### 4.7 TC-009-7: 工具调用 Span 验证

**目标**：验证 Agent 调用工具时产生的 `agent.tool.call` span。

**步骤**：

1. 使用带有工具的 Agent（如 `weather-bot`）发送需要调用工具的消息
2. 在 Jaeger UI 中找到对应的 trace
3. 展开 `agent.run` 下的子 span

**预期结果**：
- [ ] 存在 `agent.tool.call` span
- [ ] 属性包含：
  - `xyncra.tool.name`：工具名称（如 `get_weather`）
  - `xyncra.duration_ms`：工具调用耗时
- [ ] 如果工具调用失败，span 状态为 Error，并记录了错误信息

## 6. 故障排查

### Jaeger UI 无法访问

```bash
# 检查 Jaeger 容器状态
docker compose -f deploy/docker-compose.yml ps jaeger

# 查看 Jaeger 日志
docker compose -f deploy/docker-compose.yml logs jaeger

# 确认端口映射
docker compose -f deploy/docker-compose.yml port jaeger 16686
```

### 搜不到 Trace

1. 确认 `XYNCRA_TRACING_ENABLED=true`
2. 确认 `XYNCRA_TRACING_OTLP_ENDPOINT` 指向正确的 Jaeger 地址
3. 检查 server 启动日志中是否有 OTLP 连接错误
4. 确认 `XYNCRA_TRACING_SAMPLING_RATE > 0` 或目标用户在 debug 列表中
5. Jaeger 数据有延迟，等待几秒后刷新

### Span 层级不完整

1. 检查是否所有组件都使用同一个 OTLP endpoint
2. 确认 MQ trace context 传播正常工作（W3C TraceContext header）
3. 查看 server 日志中是否有 tracing 相关错误

全部满足以下条件视为通过：

3. `agent.llm.call` span 包含完整的 token 消耗属性
4. 跨 MQ 的 trace context 传播正确（同一 trace）
5. Debug 采样策略正常工作
6. Tracing 关闭时零开销、零影响
7. 三个 docker-compose 环境的 Jaeger 均可正常访问
